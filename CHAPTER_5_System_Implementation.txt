CHAPTER 5: SYSTEM IMPLEMENTATION

5.1. Reviewing the Design Solution

The Ethio-Chain Logistics platform was designed with a multi-tier architecture comprising a backend API service, a relational database, a blockchain integration layer, and a frontend web application. The design solution addressed functional requirements for managing shipment workflows, document verification, blockchain anchoring, and role-based access control across importers, sellers, customs officials, and administrators. The design incorporated architectural principles including separation of concerns through layered architecture, decoupling of synchronous operations from asynchronous blockchain processing, and security through JWT authentication and database transactions.

During implementation, key design decisions were validated. The containerized deployment approach using Docker and Docker Compose proved essential for orchestrating multiple services locally with a single command, maintaining consistency across environments. The emphasis on stateless API endpoints enabled horizontal scaling without session affinity requirements. JWT token validation on each request maintained security while preserving scalability. The modular layering of domain logic, repositories, and use cases proved critical for integrating blockchain functionality alongside traditional database operations without cross-layer dependencies.

5.2. Deciding on the Development Tools

The selection of development tools prioritized performance, ecosystem maturity, and production deployment requirements. The backend was developed using Go 1.23 for its efficient concurrency model, minimal memory footprint, and native binary compilation. Go's goroutines proved valuable for the asynchronous blockchain anchor worker. The Gin web framework (1.10.0) provided HTTP routing, middleware support, and JSON binding with minimal boilerplate. The go-ethereum package (1.14.5) enabled direct blockchain interaction.

The frontend used Next.js 16.2.1 with React 19.2.4 and TypeScript 5, providing server-side rendering, automatic code splitting, and type safety. Tailwind CSS 4.2.2 enabled rapid UI development through utility classes.

PostgreSQL 16 was selected for reliability, ACID compliance, and constraint enforcement. The pgx driver (5.7.2) provided efficient connection pooling and prepared statements. Database migrations were managed using Goose 3.24.1, enabling version control of schema changes with reversible up and down migrations. Hardhat 2.22.19 provided a local Ethereum development network for Solidity 0.8.24 smart contracts without network latency or transaction costs.

Docker and Docker Compose enabled consistent development environments across Windows, macOS, and Linux. Multi-stage Dockerfile builds minimized image size by separating build from runtime environments. Git provided version control, while Make offered a unified interface for development operations. The development environment setup required Docker, Node.js, Go 1.23, and a text editor such as Visual Studio Code.

5.3. Developing the Solution

The implementation followed clean architecture principles with layered separation of concerns: a domain layer for business entities, a repository layer for data access abstraction, a use case layer for business logic, and a delivery layer for HTTP handlers. This modular approach enabled independent development and testing of each layer while facilitating blockchain integration.

5.3.1. Backend Implementation

The backend was organized into distinct layers using Go and the Gin framework. The domain layer defined core entities (User, Shipment, Document, AnchorJob) independent of framework and database concerns. The repository layer provided data access abstractions implementing interfaces for shipment operations. The use case layer orchestrated interactions between repositories and domain objects, implementing authentication, shipment state transitions, document processing, and blockchain anchoring. The delivery layer provided HTTP handlers receiving requests, delegating to use cases, and returning JSON responses.

Authentication implemented JWT token generation and validation with role-based access control for importers, sellers, customs officials, and administrators. Passwords were hashed using bcrypt. JWT middleware validated bearer tokens on protected endpoints, verifying signatures and extracting user identity for authorization. The shipment use case managed a state machine enforcing valid transitions from INITIATED to DOCS_UPLOADED to VERIFIED to APPROVED, preventing invalid state changes.

5.3.2. Database Implementation

PostgreSQL 16 provided the relational storage with schema design driven by domain models. The pgx library offered efficient connection pooling and prepared statement support. Database migrations using Goose provided version control for schema changes with reversible up and down migrations. The schema included users table (credentials, roles, KYC status), shipments table (ports, cargo characteristics, status, blockchain anchoring), shipment_documents table (original filenames, hashes, verification status), and auxiliary tables for sellers, ESL allocations, blockchain anchor jobs, and audit trails.

Foreign key constraints maintained referential integrity, check constraints enforced domain rules (positive weights), and indexes on frequently queried columns optimized query performance. Parameterized queries prevented SQL injection attacks. All operations used database transactions to ensure atomicity: either a shipment and its audit event were both created, or both were rolled back on error.

5.3.3. Document Upload and Hashing

Document management supported uploading bills of lading, commercial invoices, and letters of credit. Upon upload, the application validated file types, computed SHA-256 hashes during file reading, stored files in organized directories, and recorded metadata in the database. File hashing using crypto/sha256 produced hexadecimal representations for UI display and blockchain anchoring. The hash computation occurred before disk persistence, detecting corruption during transfer. Document verification displayed SHA-256 hashes to users, enabling independent verification by computing hashes of local copies. Audit trails recorded upload timestamps and verification status.

5.3.4. Blockchain Anchoring

The ShipmentAnchor Solidity smart contract served as a permissioned anchor store, implementing an anchorRecord function accepting shipment key, record hash, previous hash, and record type. The contract maintained mappings preventing duplicate anchoring and provided isAnchored queries. RecordAnchored events created queryable blockchain records of all successful submissions.

The Ethereum client connected to local Hardhat networks for development or external networks for production. Anchor job submission was decoupled from user operations through asynchronous worker pattern: documents or status changes queued AnchorJob records rather than immediately anchoring. A background worker goroutine polled for pending jobs at configurable intervals, submitted them to the blockchain, and marked jobs as anchored or failed. This prevented blockchain latency from affecting API responsiveness while ensuring eventual processing. Failed jobs were retried up to configurable timeouts with careful idempotency handling to prevent duplicate anchoring.

5.3.5. Frontend Implementation

The frontend used Next.js with React and TypeScript, organized into page-based routing with directories for authentication, admin console, and importer dashboard. The frontend made HTTP requests to the backend API at localhost:8080 during development, with base URL configurable through NEXT_PUBLIC_API_BASE environment variable.

Authentication implemented login and signup pages where users provided credentials and selected roles, storing JWT tokens in local storage. Subsequent API requests included tokens in Authorization headers. The importer dashboard provided workspace for creating shipments, uploading documents, and monitoring status. Forms captured origin port, destination port, cargo type, weight, volume, and optional seller ID. The shipment list displayed user shipments with status columns and detail view links.

The shipment detail view displayed documents with SHA-256 hashes, audit trails showing state transitions, and blockchain anchoring status. An upload form submitted bills of lading, invoices, and letters of credit. Manual refresh and 15-second polling enabled real-time status updates without requiring manual refresh. React hooks managed state and side effects. Tailwind CSS provided responsive design supporting desktop and mobile. The admin approval console displayed pending KYC approvals with approve/deny buttons. Sellers could register and submit business documents for verification.

5.3.6. Integration and Deployment

Multi-stage Docker builds separated build processes from runtime environments, reducing image size by excluding compilers. The runtime image based on Alpine Linux contained only the compiled binary and minimal OS utilities. Dockerfile stages resolved dependencies, compiled code, and prepared migrations.

Docker Compose orchestrated multiple services during development and testing. The postgres service provided the database with health checks. The migrations service ran Goose migrations as an initialization step, waiting for database readiness before running. The backend service ran the Go application in a golang:1.23-alpine container with volumes mounted for live code reloading.

Environment variables in a .env file configured database credentials, JWT secrets, blockchain RPC URLs, private keys, and feature flags. This approach enabled deploying identical container images to different environments without rebuilds. Configuration managed the POSTGRES_DB, POSTGRES_USER, POSTGRES_PASSWORD, DATABASE_URL, API_PORT, JWT_SECRET, UPLOAD_DIR, BLOCKCHAIN_ENABLED, BLOCKCHAIN_RPC_URL, BLOCKCHAIN_PRIVATE_KEY, and ANCHOR_CONTRACT_ADDRESS.

The application implemented graceful shutdown by handling SIGTERM and SIGINT signals, allowing in-flight requests to complete and connections to close cleanly. Database connection pools were configured with appropriate timeouts and limits. The blockchain worker could be disabled for deployments without blockchain requirements.

Challenges during integration included managing containerized database migrations (solved with a separate migrations service waiting for database health), connecting to external Ethereum endpoints from containers (solved with host.docker.internal for host access), and managing asynchronous blockchain anchoring reliably (solved with async worker pattern and idempotent job processing). The architecture enabled developers to run the complete system locally through "make up", reducing setup complexity and enabling rapid iteration. Container image promotion through environments ensured consistency and repeatability in deployment procedures.
