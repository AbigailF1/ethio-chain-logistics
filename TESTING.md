# Testing Strategy

This suite evaluates the implemented Ethio-Chain Logistics prototype. It is not a
national-scale production certification suite. The focus is local/containerized
testing, simulated logistics data, local Ethereum-compatible blockchain tests,
and controlled user workflow evidence for Chapter Six.

## Test Categories

| Category | Command | Purpose |
| --- | --- | --- |
| Backend unit/API | `make backend-test` | Validates health routing, JWT/RBAC middleware, usecase validation, and blockchain worker behavior. |
| Blockchain contract | `make blockchain-test` | Validates Solidity anchoring, ownership, duplicate prevention, and invalid input rejection. |
| Frontend unit | `make frontend-test` | Validates UI state, upload behavior, API client behavior, and proof-badge rendering. |
| E2E evidence | `make chapter-six-evidence` | Captures Playwright screenshots for Chapter Six evidence. |
| Combined smoke | `make test` | Runs backend, blockchain, and frontend automated tests. |

## Chapter Six Evidence

Use `frontend/test-artifacts/chapter-six/` for screenshots generated by Playwright.
The login evidence test requires seeded prototype credentials:

```bash
E2E_IMPORTER_EMAIL=importer@example.com \
E2E_IMPORTER_PASSWORD=password \
make chapter-six-evidence
```

Recommended evidence to collect before finalizing Chapter Six:

- successful login/dashboard screenshot
- shipment creation screenshot
- document upload/hash screenshot
- audit timeline screenshot
- Hardhat test output screenshot
- Docker Compose services/health screenshot
- API response showing anchor status and transaction hash when blockchain is enabled
- short manual timing table for API response time, polling interval, and anchoring delay

## Prototype Limits

Treat uptime, external stakeholder integration, real GPS tracking, SMS/email
delivery, and national-scale load as design/prototype observations unless a
production environment is explicitly deployed and measured.
